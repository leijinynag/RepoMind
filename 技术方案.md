# GitHub AI Agent — 技术方案

---

## 一、项目目标

构建一个能够理解 GitHub 项目结构并支持代码级问答的 AI Agent 系统。用户输入任意公开 GitHub 仓库地址，系统自动 clone、解析、索引代码，然后通过对话界面回答关于该仓库的任意技术问题。

**核心价值主张：**
- 不只是"搜索"，而是有规划、有推理、有工具调用能力的 Agent
- RAG + Tool Use 双引擎驱动，覆盖语义搜索和精确文件读取两种场景
- 可视化 Agent 思考过程，让 AI 推理过程透明可观测
- 全部核心逻辑自主实现，不依赖 LangChain 等封装框架，具备更强的可控性和可扩展性

## 二、核心功能需求

### 2.1 Repo 管理模块

| 功能 | API | 说明 |
|------|-----|------|
| 加载 Repo | `POST /api/repo/load` | 触发 clone → 解析 → 索引全流程 |
| Repo 列表 | `GET /api/repo/list` | 返回已加载的仓库列表及状态 |
| 删除 Repo | `DELETE /api/repo/:repoId` | 清理 clone 目录 + 向量数据 |
| 获取文件树 | `GET /api/repo/:repoId/tree` | 返回目录结构 JSON |

**加载流程：**
```
输入 repo URL
    ↓
参数校验（是否合法 GitHub URL）
    ↓
git clone（shallow clone，--depth=1）
    ↓
解析目录结构 + 识别语言 / 框架
    ↓
代码切分 + Embedding 生成
    ↓
写入 Vector DB + MongoDB
    ↓
返回 repoId，状态: ready
```

### 2.2 代码索引（RAG）

**切分策略：**
- 以**函数 / 类**为粒度（优先），辅以固定 token 窗口兜底（512 token，overlap 64）
- 每个 chunk 携带元数据：`file_path`、`start_line`、`end_line`、`language`、`repo_id`

**Embedding 流程：**
```
代码文件
    ↓
语言感知解析（tree-sitter 提取函数/类块）
    ↓
生成 chunk（携带元数据）
    ↓
批量调用 Embedding API（OpenAI text-embedding-3-small）
    ↓
写入 ChromaDB（collection 按 repo_id 隔离）
```

**查询流程：**
```
用户问题
    ↓
question → embedding
    ↓
ChromaDB similarity search（top-k=8）
    ↓
返回 [code_chunk + file_path + score]
```

**MongoDB 存储结构（repo 元数据）：**
```json
{
  "repo_id": "uuid",
  "url": "https://github.com/xxx/project",
  "name": "project",
  "status": "ready | indexing | error",
  "language": "TypeScript",
  "file_count": 120,
  "indexed_at": "2024-01-01T00:00:00Z",
  "local_path": "/repos/project"
}
```

### 2.3 Agent 系统（核心）

采用经典 **ReAct（Reason + Act）** 模式，Agent Loop 完全自主实现：

```
User Question
    ↓
构建 system prompt（注入 repo 上下文 + 可用 tool 列表）
    ↓
[Thought] LLM 分析问题，决定下一步
    ↓
[Action] 解析 LLM 输出，提取工具名 + 参数
    ↓
[Tool Execution] ToolRegistry 分发执行，获取结果
    ↓
[Observation] 将结果追加到 messages 数组
    ↓
判断是否有足够信息 → 是则输出 Final Answer，否则继续循环
```

**自实现 ReAct 核心代码结构（约 120 行）：**
```typescript
// agent/AgentRunner.ts
class AgentRunner {
  private tools: ToolRegistry;
  private llm: LLMClient;
  private maxSteps = 8;

  async run(question: string, repoId: string, emitter: SSEEmitter) {
    const messages: Message[] = [
      { role: 'system', content: buildSystemPrompt(repoId, this.tools.schemas()) },
      { role: 'user', content: question }
    ];

    for (let step = 0; step < this.maxSteps; step++) {
      const response = await this.llm.chat(messages);
      const parsed = parseReActOutput(response); // 解析 Thought/Action/Input

      emitter.emit('thought', parsed.thought);

      if (parsed.type === 'final_answer') {
        emitter.emit('answer', parsed.content);
        break;
      }

      emitter.emit('action', { tool: parsed.action, input: parsed.input });
      const result = await this.tools.execute(parsed.action, parsed.input, repoId);
      emitter.emit('observation', result);

      messages.push({ role: 'assistant', content: response });
      messages.push({ role: 'user', content: `Observation: ${result}` });
    }
  }
}
```

**关键设计：**
- 最大循环轮数限制（默认 8 轮），防止无限循环
- 每一步的 Thought / Action / Observation 实时 SSE 推送给前端（可视化 Agent 思考状态）
- `parseReActOutput` 纯文本解析，不依赖 function calling 格式，LLM 无关
- 支持 early stop：解析到 `Final Answer:` 关键词时终止 loop

### 2.4 Tool 系统

| Tool | 参数 | 描述 |
|------|------|------|
| `clone_repo` | `url: string` | clone 指定 GitHub 仓库 |
| `search_code` | `query: string, top_k?: number` | 向量搜索相关代码片段 |
| `read_file` | `repo_id: string, file_path: string` | 读取指定文件完整内容 |
| `analyze_repo` | `repo_id: string` | 分析仓库目录结构 + 依赖 |
| `list_files` | `repo_id: string, dir?: string` | 列出目录下的文件 |
| `grep_code` | `repo_id: string, pattern: string` | 正则搜索代码（精确查找符号/变量名） |

**自实现 ToolRegistry：**
```typescript
// tools/ToolRegistry.ts
class ToolRegistry {
  private tools: Map<string, Tool> = new Map();

  register(tool: Tool) {
    this.tools.set(tool.name, tool);
  }

  schemas(): ToolSchema[] {
    return [...this.tools.values()].map(t => t.schema);
  }

  async execute(name: string, input: Record<string, unknown>, repoId: string) {
    const tool = this.tools.get(name);
    if (!tool) throw new Error(`Unknown tool: ${name}`);
    return tool.execute(input, repoId);
  }
}

// 每个 Tool 实现同一个接口，注册即可使用，无需框架
interface Tool {
  name: string;
  schema: ToolSchema;
  execute(input: Record<string, unknown>, repoId: string): Promise<string>;
}
```

**Tool Schema（注入 system prompt 格式）：**
```
Tool: search_code
Description: 在已加载的代码仓库中进行语义搜索，返回最相关的代码片段
Parameters:
  - query (string, required): 自然语言搜索描述
  - top_k (number, optional): 返回结果数量，默认 5

使用方式：
Action: search_code
Action Input: {"query": "router implementation", "top_k": 5}
```

### 2.5 Chat 系统

- **协议**：SSE（Server-Sent Events）流式输出
- **上下文管理**：每个对话 session 维护消息历史，超长时自动截断（保留最近 10 轮）
- **RAG 注入**：每次提问前先检索相关代码片段，注入 system prompt

**Chat API：**
```
POST /api/chat
Body: { repoId, sessionId, message }
Response: SSE stream

事件类型：
  event: thought      → Agent 正在思考（含 thought 文本）
  event: action       → Agent 调用工具（含 tool name + input）
  event: observation  → 工具执行结果（含摘要）
  event: token        → LLM 输出 token（流式）
  event: done         → 回答完成
  event: error        → 执行出错
```

## 七、关键技术决策

### 7.1 为什么全部自己实现而不用 LangChain 等框架？

- **可控性**：LangChain 封装层深，出问题难以调试；自实现每一行都清楚
- **可扩展性**：自建 `ToolRegistry` 和 `AgentRunner`，添加新 Tool 只需实现 `Tool` 接口并注册，无框架约束
- **学习深度**：手写 ReAct Loop 对 Agent 原理的理解远超调用封装 API
- **体积与性能**：不引入 LangChain 的庞大依赖树，冷启动更快
- **实现成本低**：核心 ReAct Loop 约 120 行，RAG Pipeline 约 200 行，完全可控

### 7.2 Embedding 选型

- **推荐 OpenAI text-embedding-3-small**：性价比高，1536 维，效果好
- **备选 bge-m3**（通过 Ollama 本地运行）：无 API 费用，但需要接受较慢速度

### 7.3 代码切分粒度

- **函数级 > 文件级**：避免超长文件导致 embedding 质量下降
- 对于无法解析的文件（如配置文件、Markdown），退回到固定 512 token 滑动窗口
- tree-sitter 支持 JS/TS/Python/Go/Rust 等主流语言，覆盖大多数 GitHub 项目

### 7.4 SSE vs WebSocket

- 选 **SSE**：单向流式输出场景完全够用，实现简单，HTTP 友好，无需额外依赖

### 7.5 ReAct 解析策略

不使用 OpenAI Function Calling，而是用**纯文本格式**约束 LLM 输出：
```
Thought: 我需要先搜索路由相关代码
Action: search_code
Action Input: {"query": "router implementation"}
```
优点：LLM 无关，任何模型（DeepSeek/豆包/Ollama）均可使用，不绑定 OpenAI API 格式。

## 八、可扩展性设计

这是项目后续可以继续深化的方向，每一个都能成为独立的功能亮点：

### 8.1 增量索引（Incremental Indexing）

当前每次加载 repo 都全量重建索引，可扩展为：
- 记录每个文件的 `last_commit_hash`
- `POST /api/repo/sync` 触发增量更新，只对变更文件重新 embed
- 大型仓库（数千文件）索引时间从分钟级降至秒级

### 8.2 私有 Repo 支持

- 支持传入 `GitHub Personal Access Token`，clone 私有仓库
- Token 加密存储在 MongoDB，不落日志
- 可扩展为 OAuth 登录（GitHub App），一键授权

### 8.3 多仓库对比分析

扩展 `analyze_repo` Tool，支持跨 repo 对比：
- "这两个项目的架构有什么区别？"
- "哪个项目的测试覆盖率更高？"
- Agent 同时持有两个 repo 的上下文，进行对比推理

### 8.4 代码执行沙箱（Code Runner Tool）

新增 `run_code` Tool：
- 在 Docker 沙箱中执行代码片段（Node.js / Python）
- Agent 可以验证自己给出的代码示例是否能运行
- 用途：自动生成单元测试 + 运行验证

### 8.5 GitHub Issues / PR 分析

扩展 Tool 集，接入 GitHub REST API：
- `get_issues` Tool：获取 issue 列表，Agent 可以回答"这个项目有什么已知 Bug？"
- `get_pr_diff` Tool：读取 PR diff，Agent 可以做代码 Review
- `get_commits` Tool：分析提交历史，回答"这个功能是什么时候加的？"

### 8.6 Agent 记忆系统（Long-term Memory）

当前 Agent 每次对话都是无状态的，可扩展为：
- 将历史有价值的 Observation（如"该项目使用 Fastify 框架"）存入 MongoDB
- 下次对话前自动检索相关记忆，注入 system prompt
- 避免重复调用相同 Tool，提升响应速度

### 8.7 Webhook 自动同步

- 在 GitHub 仓库设置 Webhook，推送到 `POST /api/webhook/github`
- 每次 push 自动触发增量索引，始终保持代码索引最新
- 配合 ngrok / 公网部署使用

### 8.8 插件化 Tool 扩展

当前 `ToolRegistry` 已经是插件架构，可进一步支持：
- 动态加载 `tools/` 目录下的文件，无需重启服务即可添加新工具
- 未来可做成类似 VSCode 插件市场的形式，社区贡献 Tool

### 8.9 对话导出 & 分享

- `GET /api/session/:sessionId/export` 导出完整对话为 Markdown / PDF
- 生成分享链接，他人可只读查看对话（含 Agent 思考过程）

## 十、项目亮点总结

| 亮点 | 技术深度 |
|------|----------|
| 自实现 ReAct Agent Loop | 手写 Thought/Action/Observation 循环，不依赖任何 AI 框架，完全可控 |
| 插件式 ToolRegistry | 统一 Tool 接口，注册即可用，易于扩展新能力 |
| 函数级代码索引 | tree-sitter AST 解析，语义切分质量远高于滑动窗口 |
| Agent 思考可视化 | 实时 SSE 推送每一步推理，Framer Motion 时间线动画 |
| LLM 无关架构 | 纯文本 ReAct 格式，DeepSeek / 豆包 / Ollama 均可无缝切换 |
| 全栈 TypeScript | 前后端类型共享，接口类型安全 |
| 高可扩展性 | 增量索引、私有 Repo、GitHub API、代码沙箱等方向均可平滑接入 |
